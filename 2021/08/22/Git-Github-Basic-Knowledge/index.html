4. Git architecture

working directory (the work here is not tracked) –add– staging index (the work here is about to be committed) –commit– repository (the work here is tracked)

The numbers for each commit is Hash value or SHA-1 value.

Question:

  1. If a bad actor tampers with a file in an old version of a repository, the changed checksum would generate an error on access of that version or a descendant.
  2. If you view the content of .git/HEAD (this is a file) and find the contents ‘ref: refs/heads/new’(this is the record in that file), it means ‘HEAD currently points to a commit in branch new.’ If you checkout another branch, the contents of .git/refs/heads/new will not change, but the .git/HEAD contents will.

5. Make changes to files

  • Check status: After the commit and before editing add command, ‘git status’ will show nothing in the working tree.

  • Red Name: After editing without adding, ‘git status’ shows the red file names (means not tracked), which don’t have any operation. New file, or The file is different from what was tracked, and it is edited but not tracked.

  • Green Name: After adding the file, ‘get status’ shows the file name green (tracked in staging index)

  • Head movement info: in ‘get log’, BRANCH: (HEAD -> branchName) shows the located branch name, and FILE LOCATION: the line that info locates shows which document is being pointed.

  • Check the changes in the working area: after this command, the file in the working place will be compared with in the staging index. ‘-‘shows what is deleted, and ‘+’ shows what is added.
